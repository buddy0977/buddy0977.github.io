Family Tree — Prolog

Prolog 
/* Facts */
parent_of(helen, lily).
parent_of(oliver, james).
parent_of(sophie, james).
parent_of(jess, simon).
parent_of(ali, simon).
parent_of(lily, harry).
parent_of(james, harry).

/* Rules */
father_of(X,Y) :- male(X), parent_of(X,Y).
mother_of(X,Y) :- female(X), parent_of(X,Y).

grandfather_of(X,Y) :- male(X), parent_of(X,Z), parent_of(Z,Y).
grandmother_of(X,Y) :- female(X), parent_of(X,Z), parent_of(Z,Y).

sister_of(X,Y) :-
    female(X),
    father_of(F,Y), father_of(F,X), X \= Y.
sister_of(X,Y) :- female(X),
    mother_of(M,Y), mother_of(M,X), X \= Y.

aunt_of(X,Y) :- female(X),
    parent_of(Z,Y), sister_of(Z,X), !.

brother_of(X,Y) :-
    male(X),
    father_of(F,Y), father_of(F,X), X \= Y.
brother_of(X,Y) :- male(X),
    mother_of(M,Y), mother_of(M,X), X \= Y.

uncle_of(X,Y) :-
    parent_of(Z,Y), brother_of(Z,X).

ancestor_of(X,Y) :- parent_of(X,Y).
ancestor_of(X,Y) :- parent_of(X,Z), ancestor_of(Z,Y).
// Sample Queries
Q1
?- parent_of(lily, harry).→ true. (Lily is a parent of Harry)
Q2
?- grandfather_of(oliver, harry).→ true. (Oliver → James → Harry)
Q3
?- grandmother_of(X, harry).→ X = helen ; X = sophie.
Q4
?- ancestor_of(helen, harry).→ true. (Helen → Lily → Harry)
Q5
?- uncle_of(X, harry).→ X = simon.
Theory Family Tree — Key Concepts

Prolog is a logic programming language in which a program consists of facts and rules that together form a knowledge base, queried through a built-in inference engine using backward chaining. Facts such as parent_of(helen, lily) assert unconditional ground truths, while rules use the :- operator (read as "if") to derive new relationships from existing ones. Variables in Prolog are uppercase identifiers like X and Y, which get unified — bound to actual values — during resolution, whereas lowercase atoms like helen represent fixed constants. Derived predicates like father_of and mother_of combine the base parent_of fact with gender checks, building more specific relationships from simpler ones. The ancestor_of rule demonstrates recursion: its base case matches a direct parent, and its recursive case chains through intermediate nodes, allowing Prolog to traverse arbitrarily long ancestry paths automatically. Multiple clauses for the same predicate, as seen in sister_of, represent disjunction — a sibling relationship is satisfied through either a shared father or a shared mother, and Prolog tries each clause in order via backtracking. The inequality constraint \= prevents trivial self-matches, ensuring a person cannot be their own sibling. The cut operator ! in aunt_of prunes the search tree after the first valid answer is found, preventing duplicate results in cases where only one solution is needed. When querying something like grandmother_of(X, harry), Prolog uses backtracking to systematically find and return all possible bindings of X, making it naturally suited for querying relational family databases.

Breadth-First Search — Prolog

Prolog 
%connected(+Start, +Goal)
connected(1,7).   connected(1,8).   connected(1,3).
connected(7,4).   connected(7,20).  connected(7,17).
connected(8,6).   connected(3,9).   connected(3,12).
connected(9,19).  connected(4,42).  connected(20,28).
connected(17,10).

connected2(X,Y) :- connected(X,Y).
connected2(X,Y) :- connected(Y,X).

next_node(Current, Next, Path) :-
    connected2(Current, Next),
    not(member(Next, Path)).

breadth_first(Goal, Goal, _, [Goal]).
breadth_first(Start, Goal, Visited, Path) :-
    findall(X,
        (connected2(X,Start), not(member(X,Visited))),
        [T|Extend]),
    write(Visited), nl,
    append(Visited, [T|Extend], Visited2),
    append(Path, [T|Extend], [Next|Path2]),
    breadth_first(Next, Goal, Visited2, Path2).
// Sample Queries
Q1
?- breadth_first(1, 28, [1], Path).→ Finds shortest path from node 1 to node 28
Q2
?- connected2(7, X).→ X = 1 ; X = 4 ; X = 20 ; X = 17.
Q3
?- breadth_first(1, 42, [1], Path).→ Path goes 1 → 7 → 4 → 42
Q4
?- next_node(3, X, [1,3]).→ X = 9 ; X = 12.
Q5
?- breadth_first(1, 19, [1], Path).→ Path: 1 → 3 → 9 → 19
Theory BFS — Key Concepts

Breadth-First Search (BFS) is a graph traversal algorithm that explores nodes level by level, visiting all nodes at depth 1 before any node at depth 2, ensuring the shortest path is found in an unweighted graph. In this Prolog implementation, the graph is defined using connected facts that represent one-directional edges, and the connected2 predicate makes the graph bidirectional by providing two clauses — one matching the original direction and one reversing the arguments. The core of the BFS expansion step is findall/3, which collects all unvisited neighbours of the current node into a single list in one call, effectively simulating the queue-filling operation performed at each level. The Visited list acts as the explored set, and the not(member(X, Visited)) check ensures no node is processed more than once, preventing infinite loops in cyclic graphs. The base case breadth_first(Goal, Goal, _, [Goal]) terminates the recursion cleanly when the current node matches the goal, at which point the path list is returned and the recursion unwinds. The next_node predicate provides a helper for checking valid unvisited transitions from any given node, useful for testing graph connectivity independently. The append/3 predicate serves a dual role: it merges the current visited set with newly discovered neighbours to form Visited2, and simultaneously manages the traversal queue by appending new nodes to the path. BFS requires O(bd) space where b is the branching factor and d is the solution depth, making it more memory-intensive than DFS, but this cost guarantees optimality. The write(Visited) call at each recursive step prints the visited list at each expansion stage, making the level-by-level progress of the algorithm visible during execution.

Depth-First Search — Prolog

Prolog 
solve(Node, Solution) :-
    depthfirst([], Node, Solution).

depthfirst(Path, Node, [Node | Path]) :-
    goal(Node).
depthfirst(Path, Node, Sol) :-
    s(Node, Node1),
    \+ member(Node1, Path),
    depthfirst([Node | Path], Node1, Sol).

depthfirst2(Node, [Node], _) :-
    goal(Node).
depthfirst2(Node, [Node | Sol], Maxdepth) :-
    Maxdepth > 0,
    s(Node, Node1),
    Max1 is Maxdepth - 1,
    depthfirst2(Node1, Sol, Max1).

/* Goals and State-Space */
goal(f).    goal(j).
s(a,b).    s(a,c).
s(b,d).    s(b,e).
s(c,f).    s(c,g).
s(d,h).    s(e,i).    s(e,j).
// Sample Queries
Q1
?- solve(a, Solution).→ Solution = [f, c, a]
Q2
?- depthfirst2(a, Path, 3).→ Path = [a, b, d, h] or [a, c, f]
Q3
?- goal(X).→ X = f ; X = j.
Q4
?- depthfirst([], a, Sol).→ Sol = [f, c, a]
Q5
?- depthfirst2(a, Path, 4).→ Path = [a, b, e, j]
Theory DFS — Key Concepts

Depth-First Search (DFS) is a graph traversal strategy that explores as deep as possible along each branch before backtracking to try alternative paths, and in Prolog it maps directly onto the language's own recursive and backtracking execution model. The state-space is defined purely through the s/2 successor relation and the goal/1 predicate, making the search algorithm completely problem-independent — only these two predicates need to change for a different problem. The solve/2 predicate acts as a clean public interface, initialising the visited path as an empty list and delegating all work to depthfirst/3, which carries the accumulator. The Path accumulator records all nodes on the current active branch, and the check \+ member(Node1, Path) ensures the algorithm never re-enters a node already on the path, preventing infinite loops in cyclic graphs. DFS has O(b·m) space complexity, where b is the branching factor and m is the maximum depth, making it far more memory-efficient than BFS since only the nodes on the current path are held in memory at any time. The depth-limited variant depthfirst2 adds a Maxdepth counter that decrements with each recursive call; when it reaches zero the branch is abandoned, preventing the algorithm from descending into arbitrarily deep or infinite paths. This depth-limited DFS is the building block of Iterative Deepening DFS (IDDFS), which repeatedly runs depth-limited searches with increasing limits to achieve BFS-level completeness with DFS-level memory usage. In this program two goal states f and j are defined, so the same search infrastructure can find either goal depending on which is encountered first in the tree traversal. Without both the visited-node check and an optional depth cap, DFS would be incomplete and potentially non-terminating on graphs with cycles or infinite branches.

Hill Climbing — Python

Python 
def f(x):
    return -x*x + 5

x = 2.0
step = 0.1
iterations = 0

for i in range(100):
    left  = x - step
    right = x + step
    if f(left) > f(x):
        x = left
        iterations += 1
    elif f(right) > f(x):
        x = right
        iterations += 1
    else:
        print(f"Algorithm converged after {iterations} iterations.")
        break

if abs(x) < 1e-10:
    x = 0.0

print("Best solution =", x)
print("Maximum value =", round(f(x), 4))
Theory Hill Climbing — Key Concepts

Hill Climbing is a local search optimisation algorithm that starts from an initial solution and iteratively moves to the neighbouring state with the highest objective value, continuing until no neighbour offers improvement. It belongs to the family of greedy algorithms because it always commits to the best immediate move without any lookahead or memory of previously visited states. The objective function in this program is f(x) = -x² + 5, a downward-opening parabola with a single global maximum at x = 0 where f(0) = 5, making it an ideal and easily verifiable test case. At each iteration, two candidate neighbours are evaluated — x - step (left) and x + step (right) — and the algorithm moves to whichever improves the function value; if neither does, it concludes that a local optimum has been reached and terminates early via break. The step size of 0.1 controls the balance between precision and speed: a larger step covers the search space faster but may overshoot the exact peak, while a smaller step gives finer resolution at the cost of more iterations. A key limitation of Hill Climbing is that it can permanently get stuck at local maxima, plateaus, or ridges in more complex multi-modal functions, since it has no mechanism to escape once no immediate improvement is visible. In this specific case the function has only one peak so the algorithm always converges to the global optimum, but this would not be guaranteed on a function with multiple local maxima. The near-zero floating-point check abs(x) < 1e-10 handles the fact that repeated floating-point arithmetic may leave x as a very small non-zero number rather than exactly 0.0, snapping it to the true optimum for clean output. Variants such as Stochastic Hill Climbing, Random-Restart Hill Climbing, and Simulated Annealing were developed to overcome the local-maxima problem by introducing randomness, multiple starting points, or occasional downhill moves respectively.

Genetic Algorithm — Python

Python 
import random

TARGET = 20

# Fitness Function
def fitness(x):
    return -abs(TARGET - x)

# Initial Population
population = [random.randint(0, 30) for _ in range(6)]

for generation in range(10):
    # Selection (best 2 individuals)
    population.sort(key=fitness, reverse=True)
    parent1 = population[0]
    parent2 = population[1]

    # Crossover
    child = (parent1 + parent2) // 2

    # Mutation
    if random.random() < 0.3:
        child += random.choice([-1, 1])

    # Replace worst individual
    population[-1] = child
    print("Generation", generation + 1, "Best =", parent1)

print("\nBest Solution =", population[0])
Theory Genetic Algorithm — Key Concepts

A Genetic Algorithm (GA) is a population-based metaheuristic optimisation technique inspired by Darwinian natural selection, where a population of candidate solutions evolves over successive generations through selection, crossover, and mutation until an optimal or near-optimal solution is found. The algorithm begins by generating an initial population of six random integers in the range 0–30, each representing a candidate solution to the problem of reaching the target value of 20. The fitness function -abs(TARGET - x) scores each individual based on its distance from the target, returning 0 for a perfect match and increasingly negative values for individuals that are further away, giving the selection process a clear directional signal. In each generation, the population is sorted by fitness in descending order and the top two individuals are selected as parents — a strategy known as truncation selection, directly embodying the survival-of-the-fittest principle. Crossover (recombination) then creates a new child by computing the integer average of the two parents: (parent1 + parent2) // 2, which mimics biological gene mixing and tends to produce offspring that inherit the better traits of both parents. Mutation is applied with a 30% probability by randomly nudging the child by ±1; without this step the population would rapidly converge to a narrow region of the search space and lose the diversity needed to explore other areas. Elitism is implicitly enforced by always replacing the lowest-ranked individual with the new child, ensuring the best solutions already found are never discarded and the population quality is monotonically non-decreasing. Unlike Hill Climbing, which operates on a single candidate solution and is prone to getting stuck at local optima, GAs maintain a diverse population throughout, significantly reducing the risk of premature convergence. The three key parameters — population size, mutation rate, and number of generations — control the exploration-exploitation trade-off: larger values improve solution quality at the cost of additional computation time.
